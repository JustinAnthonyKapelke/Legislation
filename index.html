Udgiv ikke sårbarheder.

Cyber Resilience Act er på vej – og den gælder jeres produkt.

CRA er ikke valgfri. Den kan blokere jeres release.

Hvad betyder det?

Cyber Resilience Act er en EU-lov, der stiller krav til sikker software og produkter.

Hvis I ikke overholder den → kan I ikke sælge i EU.

Det her er ikke bare noget “compliance”. Det handler om, hvorvidt jeres produkt overhovedet må være på markedet.

6 krav I ikke kan ignorere

1. Security by Design

  • Tænk sikkerhed før I koder
  • Lav threat modeling i designfasen
  • Ikke noget man “lige tilføjer bagefter”
  • Dokumentér jeres valg

2. Håndtering af sårbarheder

  • Find og fix sårbarheder
  • Varsling inden for 24 timer
  • Fuld rapport inden for 72 timer
  • “Vi fixer det senere” er ikke en mulighed

3. Secure by Default

  • Produktet skal være sikkert fra start
  • Ingen svage standard passwords
  • Sikkerhed skal være slået til
  • Ingen manuel hardening nødvendig

4. Dependency styring

  • Overblik over alle libraries (SBOM)
  • Scan automatisk for risici
  • Ja, ting som Log4j er præcis det CRA går efter
  • Kend jeres supply chain

5. Risikovurdering

  • Identificér trusler systematisk
  • Både i design og udvikling
  • Dokumentér hvordan I håndterer dem
  • Det er en løbende proces

6. Dokumentation

  • Klar til audit
  • Designbeslutninger dokumenteret
  • Vedligehold en risk log
  • Bevis for CE-mærkning

Reality check

Hvilken pipeline kører I?

Feature → Ship → Bug → Ignorer → Breach ❌

eller

Threat model → Sikker kode → Test → Ship sikkert → Monitor ✓

Konsekvenser

Hvis I ikke overholder det

  • Bøder op til €15 mio. eller 2,5% af omsætning
  • Produktet må ikke sælges i EU
  • Tab af indtjening
  • Skadet omdømme

Hvis I gør

  • Adgang til hele EU-markedet
  • Mere tillid fra kunder
  • Hurtigere reaktion på incidents
  • Færre problemer efter release
